# Tagwise Efficiency & Code Quality Improvements

## Group 1: Bug Fix — Path Encoding Corruption (Effort: S)

**Problem:** `rootToFilename`/`filenameToRoot` in `src/repositories/library.repository.ts:42-51` replaces ALL underscores with dots, corrupting any path that contains an underscore (e.g., `/home/user/My_Music` becomes `/home/user/My.Music`).

**Fix:**
- Use `btoa`-style Base64 encoding or `encodeURIComponent` for the filename mapping
- `rootToFilename`: strip leading `/`, replace `/` with `__`, then use Base64URL encoding for the rest
- `filenameToRoot`: decode Base64URL, replace `__` with `/`, prepend `/`
- Add a **cache migration**: detect old-format filenames (no Base64) and re-encode them on first `listRoots()` call

**Files:** `src/repositories/library.repository.ts`

## Group 3: Performance — Parallelize Async Operations (Effort: M)

**Problem:** Several independent async operations run sequentially, wasting time.

**Changes:**

| Location | Current | Fix |
|----------|---------|-----|
| `app-provider.tsx:93-95` | `for await` over producers | `Promise.allSettled(selected.map(p => producerService.run(p, folder)))` |
| `use-operations.tsx:207-214` | Serial `filesystemService.rename()` | Batch with `Promise.allSettled()` (or a concurrency-limited mapper for >50 files) |
| `cache.service.ts:16-31` | Serial `libraryRepository.load()` in `list()` | `Promise.all(roots.map(r => this.libraryRepository.load(r)))` |

**Files:** `src/presentation/tui/context/app-provider.tsx`, `src/presentation/tui/hooks/use-operations.tsx`, `src/services/cache.service.ts`

## Group 4: Performance — Filesystem Service Improvements (Effort: M)

**Problem:** `filesystem.service.ts` has two issues: (a) `existsSync` in an async method blocks the event loop, (b) `hasChangedSince` does sequential `stat()` per file.

**Fix:**
- Replace `existsSync` with `access` from `node:fs/promises` (catch `ENOENT` → false)
- In `hasChangedSince`, batch `stat()` calls with `Promise.all()` per directory level, short-circuit on first `mtime > since`
- Consider checking directory `mtime` first before recursing into subdirectories

**Files:** `src/services/filesystem.service.ts`

## Group 5: Performance — Compact Cache JSON (Effort: S)

**Problem:** Cache files use `JSON.stringify(data, null, 2)` (pretty-printed), wasting disk space and parse time.

**Fix:** Change to `JSON.stringify(data)` for cache-only output. Cache files are machine-read-only, human readability is unnecessary.

**Files:** `src/repositories/library.repository.ts:74`

## Group 6: Architecture — Singleton LibraryRepository (Effort: S)

**Problem:** `LibraryRepository` is instantiated independently in 3 containers (`cache.container.ts:4`, `scanner.container.ts:5`, `producer.container.ts:8`), wasting memory and risking inconsistency.

**Fix:**
- Create a shared `src/containers/repository.container.ts` that exports a single `libraryRepository` instance
- Update `cache.container.ts`, `scanner.container.ts`, `producer.container.ts` to import from the shared container
- This also sets the stage for an in-memory cache layer (Group 7)

**Files:** `src/containers/cache.container.ts`, `src/containers/scanner.container.ts`, `src/containers/producer.container.ts`, new `src/containers/repository.container.ts`

## Group 7: Performance — In-Memory Library Cache (Effort: M)

**Problem:** The same library JSON is loaded from disk and parsed up to 3 times per user flow (`app-provider.tsx:41,57,83`).

**Fix:**
- Add a `Map<string, Library>` cache inside `LibraryRepository`
- On `load()`: check memory cache first, fall back to disk, cache the result
- On `save()`: also update the memory cache
- On `delete()`: evict from memory cache

**Files:** `src/repositories/library.repository.ts`

## Group 15: Performance — Separate Tree Structure from Status (Effort: M)

**Problem:** The tree is rebuilt from scratch on every operation status change, even though the tree topology (folders, leaf membership) only depends on file paths, not statuses.

**Fix:**
- Split `computeVisibleNodes` into two memoized parts:
  1. `buildTree(operations, root)` — depends only on operation paths (memoize with operation paths as key)
  2. Status overlay — applied during rendering, not during tree construction
- This avoids rebuilding the tree structure when only statuses change

**Files:** `src/presentation/tui/components/tree-view.tsx`, `src/presentation/tui/screens/review.tsx`
